## On the Client's Server — Pre-Flight Checks

Before running the script, open **SSMS** on the server and run these three queries to collect the values you'll need during the `.env` setup:

```sql
-- 1. Find the SAGE Stock Code ID for Registrar Services
SELECT StockLink, Description_1 FROM StkItem
WHERE Description_1 LIKE '%Registrar%' OR Description_1 LIKE '%Domain%'

-- 2. Find the default Warehouse ID
SELECT WhseLink, WhseCode, WhseDescription FROM _etblWarehouse

-- 3. Find the VAT Tax Type ID
SELECT idTaxRate, cTaxRateCode, fTaxRate FROM TaxRate
WHERE fTaxRate = 7.5

-- 4. Create the VeriSagePro database if it doesn't exist yet
CREATE DATABASE VeriSagePro;
```

Write these three numbers down. The deploy script will ask for them.

## What the Script Does — Step by Step

| Step | What happens |
|---|---|
| Checks Node.js ≥ v18 | Stops immediately with a clear message if missing |
| Checks / installs PM2 | Installs globally with `npm install -g pm2` if not found |
| Detects USB source files | Validates both backend and frontend folders exist before touching anything |
| Copies files to server | Uses `xcopy` to mirror both folders to `C:\VeriSagePro\` |
| Runs `npm install` | Installs all production dependencies into `node_modules` |
| Builds `.env` interactively | Asks each value one at a time, writes the completed file |
| Runs migration | `node src/config/migrate.js` — creates tables, creates admin user |
| Starts with PM2 | Registers the process as `verisage-pro`, saves the PM2 process list |
| Configures boot startup | `pm2 startup` — app automatically restarts after a server reboot |
| Health check | Hits `http://localhost:4000/health` and prints the response |

## How the Server Runs After Deployment

```
Windows boots
  └─ PM2 auto-starts  (registered as a Windows startup task)
       └─ node src/index.js  (the Express server)
            ├─ Port 4000 open
            ├─ GET /         → serves React frontend (index.html)
            ├─ GET /api/*    → backend REST API
            ├─ POST /api/webhook/topup  → CoCCA transactions
            └─ Schedulers running in background (retry + income recognition)
```

NiRA staff open a browser and go to `http://<server-ip>:4000`. That's it. The React dashboard loads, they log in, and everything is live. No VS Code. No terminal stays open. No separate Vite dev server.

## Day-to-Day Management After Deployment

Double-click `manage.bat` (as Administrator) from the USB or from the server itself. It gives you a numbered menu:

```
[1] Check status          [6] View last 50 log lines
[2] View live logs        [7] Open .env for editing
[3] Restart application   [8] Run migration again
[4] Stop application      [9] Check health endpoint
[5] Start application     [0] Exit
```

Or use these commands directly in any Command Prompt:

```bat
pm2 status                      :: Is the app running?
pm2 logs verisage-pro           :: Stream live logs (Ctrl+C to stop)
pm2 restart verisage-pro        :: Apply .env changes or after update
pm2 stop verisage-pro           :: Graceful stop
pm2 logs verisage-pro --lines 100 --nostream   :: Last 100 lines
```

## Updating the Application Later

When you deliver a new version:

1. Run `node obfuscate.js` on your machine → new `dist/` folder.
2. Build frontend again → new `frontend/dist/`.
3. Copy both to USB.
4. On the client server, run:
```bat
pm2 stop verisage-pro
xcopy "D:\VeriSagePro\backend\*" "C:\VeriSagePro\backend\" /s /e /y
xcopy "D:\VeriSagePro\frontend-build\*" "C:\VeriSagePro\public\" /s /e /y
cd C:\VeriSagePro\backend
npm install --omit=dev
pm2 start src/index.js --name verisage-pro
pm2 save
```

The `.env` file survives updates because `xcopy` with `/y` only overwrites files that exist on the USB — and you never put `.env` on the USB.
